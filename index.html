iOS - 监测UIViewController内存泄漏

发表于 2017-07-06

一般情况下，弹出一个新的UIViewController我们可以通过以下两个方法：

[self.navigationController pushViewController:viewController animated:YES];  

// 或者
[self presentViewController:viewController animated:YES completion:nil];

同样，对应退出这个UIViewController也有以下两个方法： 

[self dismissViewControllerAnimated:YES completion:nil];

// 或者
[self.navigationController popViewControllerAnimated:YES];

当调用了退出UIViewController的方法时，通常情况下对应的UIViewController应该立即从内存中释放，如果没有正确释放，则有可能是产生了内存泄露。

基于此，在不入侵开发代码的情况下，我们可以通过hook掉这两个方法，添加我们的监测代码，从而监测UIViewController在退出时是否得到了正确的释放。对于如何hook，可以参考之前翻译的一篇文章Method Swizzling

具体做法如下：

首先，通过Method Swizzling，为基类 NSObject 添加一个方法 -willDealloc，这方法里面用一个弱指针指向self，并在3秒后，通过这个弱指针调用 -alertIfNoDealloc，而 -alertIfNoDealloc 主要作用是弹框告诉开发人员有UIViewController内存泄漏了。

接着，我们通过Method SwizzlinghookUIViewController的dismissViewControllerAnimated:和UINavigationController的popViewControllerAnimated :,在hook方法上面调用-willDealloc,即UIViewController在调用了退出的方法后，3秒后如果弱指针为nil，则代表已经释放了，如果3秒后弱指针还在，则代表泄漏了，从而执行alertIfNoDealloc弹框，从而达到监测的目的。

代码如下： 

- (BOOL)willDealloc {  
  __weak id weakSelf = self;  
  dispatch_after(dispatch_time(DISPATCH_TIME_NOW, (int64_t)(3 * NSEC_PER_SEC)), dispatch_get_main_queue(), ^{  
        [weakSelf alertIfNoDealloc];  
   }); 
   return YES;  
}

- (void)alertIfNoDealloc {  
  UIAlertController *vc = [UIAlertController alertControllerWithTitle:@"内存泄漏提示" message:@"有内存产生了泄漏" preferredStyle:UIAlertControllerStyleActionSheet];  
  UIAlertAction *action = [UIAlertAction actionWithTitle:@"确定" style:UIAlertActionStyleDefault handler:nil];
  [vc addAction:action];
  [self presentViewController:vc animated:YES completion:nil];

}

iOS - NSProxy学习笔记

发表于 2017-05-05

Objective-C中,NSObject类作为大部分类的基类。除此之外，还有一个比较少用的抽象基类 —— NSProxy。
先来看看NSProxy类的头文件定义。

可以看到NSProxy遵循了NSObject协议，同时还有个属性是isa指针。这个isa指针跟NSObject是一样的。从名字上看，NSProxy更适合做代理转发，我们先来看看它的两个方法：

- (void)forwardInvocation:(NSInvocation *)invocation;
- (nullable NSMethodSignature *)methodSignatureForSelector:(SEL)sel;

forwardInvocation: 在你给NSProxy对象发送一个他不认识的消息时，系统会抛出异常，但在抛异常之前会先调用forwardInvocation:这个方法，同时传递一个NSInvocation对象作为消息参数，该参数包含了消息的原始封装，例如消息的实现target和消息签名。你可以重写该方法，对不能处理的消息做一些特别的处理，来避免程序崩溃，同时你也可以用来将消息转发给其它对象，让其他能实现该NSInvocation的对象去处理。

NSInvocation的消息签名是通过methodSignatureForSelector:返回的，所以系统同时会先调用methodSignatureForSelector:这个方法。该方法的参数是selector，即该方法返回的是一个selector的签名，如果该方法返回nil，系统会发出doesNotRecognizeSelector:的Exception。

一般使用NSProxy做代理转发，我们只需要重写上面的两个方法 

@interface MyProxy : NSProxy
   + (id)proxyWithTarget:(id)obj;
@end


@interface MyProxy ()
@property (nonatomic, strong) id target;
@end

@implementation MyProxy

+ (id) proxyWithTarget:(id)obj {  
    MyProxy *instance = [MyProxy alloc];
    instance.target = obj;

    return instance;
}

- (NSMethodSignature *)methodSignatureForSelector:(SEL)sel {
    return [_target methodSignatureForSelector:sel];
}

- (void)forwardInvocation:(NSInvocation *)invocation {
    if ([_target respondsToSelector:invocation.selector]) {
        NSString *selectorName = NSStringFromSelector(invocation.selector);

        NSLog(@"start calling \"%@\".", selectorName);
        [invocation invokeWithTarget: _target];
        NSLog(@"end calling \"%@\".", selectorName);
    }
}
@end

以下是测试代码 

NSString *string = (NSString *)[MyProxy proxyWithTarget:[NSString stringWithFormat:@"hello word"]];
NSInteger length = [string length];
NSLog(@"length = %ld", length);

NSArray *array = (NSArray *)[MyProxy proxyWithTarget:[NSArray array]];
NSInteger count = [array count];
NSLog(@"count = %ld", count);

输出结果如下：

2017-07-06 10:32:57.547309+0800 Test[13765:3933663] start calling "length".
2017-07-06 10:32:57.547390+0800 Test[13765:3933663] end calling "length".
2017-07-06 10:32:57.547404+0800 Test[13765:3933663] length = 10
2017-07-06 10:32:57.547452+0800 Test[13765:3933663] start calling "count".
2017-07-06 10:32:57.547476+0800 Test[13765:3933663] end calling "count".
2017-07-06 10:32:57.547488+0800 Test[13765:3933663] count = 0

可以看到，在向MyProxy对象发送消息时，它将消息转发给了实际的对象。在向实际的对象发送消息时，我们打出了该消息的方法名。NSProxy有很多用途：

  • 我们可以用它来拦截或者记录一些有趣的消息调用。
  • 解决一些场景下的循环引用 （YYLabel中的YYFPSLabel就是使用NSProxy解决了循环引用的问题）
  • 假设MyProxy有两个target，我们将消息转发到其中能接受该消息的target，从而实现OC的多继承

iOS - 成员变量和属性区别

发表于 2017-04-20

前几天在cocoaChina看到这道题，记录下。

@property 的本质是什么？
@property = ivar + getter + setter;
下面解释下：
“属性” (property)有两大概念：ivar（实例变量）、存取方法（access method ＝ getter + setter）。
“属性” (property)作为 Objective-C 的一项特性，主要的作用就在于封装对象中的数据。 Objective-C 对象通常会把其所需要的数据保存为各种实例变量。实例变量一般通过“存取方法”(access method)来访问。其中，“获取方法” (getter)用于读取变量值，而“设置方法” (setter)用于写入变量值。这个概念已经定型，并且经由“属性”这一特性而成为Objective-C 2.0的一部分。 而在正规的 Objective-C 编码风格中，存取方法有着严格的命名规范。 正因为有了这种严格的命名规范，所以 Objective-C 这门语言才能根据名称自动创建出存取方法。其实也可以把属性当做一种关键字，其表示:编译器会自动写出一套存取方法，用以访问给定类型中具有给定名称的变量。 所以你也可以这么说：
@property = getter + setter;

例如下面这个类：

@interface Person : NSObject 
@property NSString *firstName; 
@property NSString *lastName; 
@end

上述代码写出来的类与下面这种写法等效：

@interface Person : NSObject 
- (NSString *)firstName; 
- (void)setFirstName:(NSString *)firstName; 
- (NSString *)lastName; 
- (void)setLastName:(NSString *)lastName; 
@end

ivar、getter、setter 是如何生成并添加到这个类中的?
“自动合成”(autosynthesis)
完成属性定义后，编译器会自动编写访问这些属性所需的方法，此过程叫做“自动合成”(autosynthesis)。需要强调的是，这个过程由编译器在编译期执行，所以编辑器里看不到这些“合成方法”(synthesized method)的源代码。除了生成方法代码 getter、setter 之外，编译器还要自动向类中添加适当类型的实例变量，并且在属性名前面加下划线，以此作为实例变量的名字。在前例中，会生成两个实例变量，其名称分别为 _firstName与_lastName。也可以在类的实现代码里通过 @synthesize语法来指定实例变量的名字.

@implementation Person 
@synthesize firstName = _myFirstName; 
@synthesize lastName = _myLastName; 
@end

我为了搞清属性是怎么实现的,曾经反编译过相关的代码,大致生成了五个东西： 

+ 1）OBJC_IVAR_$类名$属性名称 ：该属性的“偏移量” (offset)，这个偏移量是“硬编码” (hardcode)，表示该变量距离存放对象的内存区域的起始地址有多远。  
+ 2）setter与getter方法对应的实现函数  
+ 3）ivar_list ：成员变量列表  
+ 4）method_list ：方法列表  
+ 5）prop_list ：属性列表

也就是说我们每次在增加一个属性,系统都会在ivar_list中添加一个成员变量的描述,在method_list中增加setter与getter方法的描述,在属性列表中增加一个属性的描述,然后计算该属性在对象中的偏移量,然后给出setter与getter方法对应的实现,在setter方法中从偏移量的位置开始赋值,在getter方法中从偏移量开始取值,为了能够读取正确字节数,系统对象偏移量的指针类型进行了类型强转.

Method Swizzling（译）

发表于 2017-02-01

本文译自 http://nshipster.com/method-swizzling 作者Mattt Thompson

在上周对象关联的文章中，我们探索了关于Objective-C的黑魔法–运行时。这周，我们进一步讨论关于运行时最有争议的技术：Method Swizzling。

Method Swizzling是一门改变一个已经存在的selector的实现的技术。通过这一技术，我们可以在运行时通过修改类的分发表中selector对应的函数，来修改方法的实现。

假如，我们想要追踪每一个viewcontroller的出现，就必须在每个控制器的viewDidAppear方法上面添加我们的追踪代码，这会导致代码十分的臃肿。子类化也是一种实现方式，但这样的话需要子类化UIViewController, UITableViewController, UINavigationController以及其他UIViewController的子类，同样会导致大量重复的代码。

幸运的是，我们还有另外一种方式：在分类里面使用Method Swizzling。 以下是具体实现：

#import <objc/runtime.h>

@implementation UIViewController (Tracking)

+ (void)load {
static dispatch_once_t onceToken;
dispatch_once(&onceToken, ^{
    Class class = [self class];

    SEL originalSelector = @selector(viewWillAppear:);
    SEL swizzledSelector = @selector(xxx_viewWillAppear:);

    Method originalMethod = class_getInstanceMethod(class, originalSelector);
    Method swizzledMethod = class_getInstanceMethod(class, swizzledSelector);

    // 如果想swizzling类方法，可以使用以下注释代码
    // Class class = object_getClass((id)self);
    // ...
    // Method originalMethod = class_getClassMethod(class, originalSelector);
    // Method swizzledMethod = class_getClassMethod(class, swizzledSelector);

    BOOL didAddMethod = class_addMethod(class, 
                                        originalSelector, 
                                        method_getImplementation(swizzledMethod),
                                        method_getTypeEncoding(swizzledMethod));
    if (didAddMethod) {
        class_replaceMethod(class, 
                            swizzledSelector, 
                            method_getImplementation(originalMethod), 
                            method_getTypeEncoding(originalMethod));
    } else {
        method_exchangeImplementations(originalMethod, swizzledMethod);
    }
  });
}

#pragma mark - Method Swizzling

- (void)xxx_viewWillAppear:(BOOL)animated {            
        [self xxx_viewWillAppear:animated];
        NSLog(@"viewWillAppear: %@", self);
}   

@end

上面的例子，我们通过Method Swizzling修改了UIViewController的@selector(viewWillAppear:)对应的函数指针，使其实现指向了我们自定义的xxx_viewWillAppear的实现。这样，当UIViewController及其子类的对象调用viewWillAppear时，实际上调用的是xxx_viewWillAppear，都会打印出一条日志信息。

还有许多场景可以使用method swizzling，在此不多举例。这里我们说说使用method swizzling需要注意的一些问题：

+load 和 +initialize

Swizzling应该总是在+load中实现

在运行时，每个类会自动调用两个方法。+load方法在类最初加载的时候调用，+initialize在应用程序调用该类的第一个类方法或者实例方法时调用。这两个方法都是可选的，只有在该方法被实现了才会执行。

由于Method Swizzling会影响类的全局状态，因此要避免竞争条件的可能性。+load在类初始化的时候加载，从而保证了改变系统行为的一致性。相比之下，+initialize在运行的时候没有这样的保证，事实上，如果这个类没有被发送消息，该方法永远都不会被调用。

dispatch_once

Swizzling应该总是在dispatch_once中实现

同样，由于Method Swizzling改变的是类的全局状态，我们需要在运行的时候采取一些预防措施。原子性就是这样的一种措施，能够保证代码在即使有多个线程的情况下也只会被执行一次。GCD的dispatch_once可以确保这种行为，我们应该将其视为Method Swizzling的标准做法。

Selectors, Methods, 和Implementations

在Objective-C中，选择器(Selectors)，方法(Methods)，和实现(Implementations)是运行时一些特别的点，虽然在普通的情况下，它们更多出现在消息转发的过程中。

以下是Objective-C运行时文档中对这几个术语的表述：

  1. Selector（typedef struct objc_selector * SEL）:在运行时中表示一个方法的名称。一个方法选择器是一个C字符串，它是在Objective-C运行时被注册的。选择器由编译器自动生成，并且在类被加载时通过运行时自动做映射操作。
  2. Method（typedef struct objc_method * Method）：在类定义中表示一个方法的类型
  3. Implementation（typedef id (*IMP)(id, SEL, …)）：这是一个指针类型，指向方法实现函数的开始位置。这个函数使用由当前CPU架构实现的标准C调用规范。第一个参数是指向对象自身的指针(self)，第二个参数是方法选择器，然后是方法的实际参数。

最好的理解这几个概念之间的关系：一个类在运行时中维护一个消息分发表；在分发表中的每个入口是一个方法(Method)，其中key是一个特定名称，即选择器(SEL)，其对应一个实现(IMP)，即指向底层C函数的指针。

为了swizzle一个方法，我们可以在分发表中将现有的选择器转换到不同的实现，从而将该选择器对应的原始实现转换到一个新的选择器中。

调用_cmd

我们先来看看以下的代码

- (void)xxx_viewWillAppear:(BOOL)animated {
        [self xxx_viewWillAppear:animated];
        NSLog(@"viewWillAppear: %@",NSStringFromClass([self class]));
    }

咋看上去是会导致无限循环的。但令人惊奇的是，并没有出现这种情况。在swizzling的过程中，方法中的[self xxx_viewWillAppear:animated]已经被重新指定到UIViewController类的viewWillAppear:中。在这种情况下，不会产生无限循环。不过如果在这个方法中我们调用的是[self viewWillAppear:animated]，则会产生无限循环，因为这个方法的实现在运行时已经被重新指定为xxx_viewWillAppear:了。

注意事项

Swizzling通常被称作是一种黑魔法，容易产生不可预知的行为和无法预见的后果。虽然这不是安全的做法，但如果遵从以下几点预防措施的话，还是比较安全的：

  1. 总是调用方法的原始实现（除非有更好的理由不这么做）：API提供了一个输入与输出约定，但其内部实现是一个黑盒。Swizzle一个方法而不调用原始实现可能会打破私有状态底层操作，从而影响到程序的其它部分。
  2. 避免冲突：给自定义的分类方法加前缀，从而使其与所依赖的代码库不会存在命名冲突。
  3. 明白是怎么回事：简单地拷贝粘贴swizzle代码而不理解它是如何工作的，不仅危险，而且会浪费学习Objective-C运行时的机会。阅读Objective-C 关于运行时文档参考和查看头文件以了解事件是如何发生的。
  4. 小心操作：无论我们对Foundation, UIKit或其它内建框架执行Swizzle操作抱有多大信心，需要知道在下一版本中许多事可能会不一样。

如同对象关联，Method Swizzling在你需要使用到的时候会是一门强大的技术，但是需要谨慎使用。

block未判空导致的EXC_BAD_ACCESS崩溃

我们在调用block时，如果这个block为nil，则程序会崩溃，报类似于EXC_BAD_ACCESS(code=1, address=0x10)异常[64位下的结果，如果是32位，则address=0xc]。如下图所示，这个异常表示程序在试图读取内存地址0x10的信息时出错。

在定义一个block时，编译器会在栈上创建一个结构体。

block就是指向这个结构体的指针。其中的invoke就是指向具体实现的函数指针。当block被调用时，程序最终会跳转到这个函数指针指向的代码区。而当block为nil时，程序就会试图去读取0x10地址的信息。如下所示： 

struct __block_impl {
    void *isa;      // offset:0x00
    int Flags;      // offset:0x08
    int Reserved;   // offset:0x0c
    void *FuncPtr;  // offset:0x10
};

而这个地址什么都不会有，于是抛出一个segmentation fault。所以我们在使用block时，应该先判断block是否为空。
